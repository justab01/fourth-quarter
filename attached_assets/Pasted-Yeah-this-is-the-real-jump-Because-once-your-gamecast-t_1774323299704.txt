# The real idea

ESPN’s tracker feels good because it is not just “showing live data.”
It is doing 3 things at once:

1. **Building a clean event timeline**
2. **Mapping those events to a visual field/court/rink**
3. **Updating the UI live without feeling glitchy**

That is the whole game.

---

# What you should build

## Layer 1: Canonical Live Event Engine

This is the brain.

Every live game, no matter the sport, should be normalized into one universal structure.

Example shape:

```ts
type CanonicalGameEvent = {
  id: string
  gameId: string
  sport: "basketball" | "football" | "baseball" | "soccer" | "hockey" | "mma"
  league: string
  timestamp: string
  sequence: number
  period: string
  clock?: string
  teamId?: string
  playerIds?: string[]
  eventType: string
  subType?: string
  x?: number
  y?: number
  outcome?: "made" | "missed" | "turnover" | "goal" | "save" | "penalty" | "hit"
  value?: number
  meta?: Record<string, any>
}
```

This gives you one universal feed.

Then on top of that, each sport gets translated into its own visual logic.

---

# Layer 2: Sport-specific parser adapters

Do **not** let raw ESPN data hit your UI directly.

That’ll become a nightmare.

Instead build adapters like:

* `parseBasketballEvent()`
* `parseFootballEvent()`
* `parseBaseballEvent()`
* `parseSoccerEvent()`
* `parseHockeyEvent()`
* `parseMMAEvent()`

Each adapter takes raw API/play-by-play data and outputs `CanonicalGameEvent`.

That is how you scale across all sports without your frontend turning into hell.

---

# Layer 3: Tracker state store

For every live game, you need a derived live state that updates from incoming events.

Example:

```ts
type GameTrackerState = {
  gameId: string
  sport: string
  currentPeriod: string
  clock: string
  possessionTeamId?: string
  homeScore: number
  awayScore: number
  lastEvent?: CanonicalGameEvent
  recentEvents: CanonicalGameEvent[]
  momentumWindow: CanonicalGameEvent[]
  visualState: Record<string, any>
}
```

This is the state the tracker UI actually reads from.

Not raw play-by-play.

# The architecture I’d use

## Backend

You already have:

* ESPN live APIs
* WebSocket updates
* PostgreSQL
* React Query cache invalidation

Good.

Now add these services:

### 1. Live Polling / Ingest Service

Pull live game feeds every few seconds.

Responsibilities:

* fetch play-by-play
* fetch box score
* fetch tracker/location data where available
* compare to last snapshot
* identify new events only

### 2. Event Normalizer

Convert raw feed to canonical events.

### 3. State Deriver

Take all events and compute:

* possession
* score
* momentum
* court/field state
* runners/base state
* drive state
* attack zone
* power play state
* etc.

### 4. WebSocket Broadcaster

Push only clean updates to clients.

Not full game payload every time.

Push:

* new events
* score changes
* state patch
* visual patch

---

# What your websocket payload should look like

Keep it lean.

```ts
type LiveGameUpdatePayload = {
  gameId: string
  type: "event_batch" | "score_update" | "tracker_patch" | "status_update"
  timestamp: string
  events?: CanonicalGameEvent[]
  trackerPatch?: Partial<GameTrackerState>
}
```

That way your UI patches state instead of rebuilding everything.

# The real backend challenge

Your biggest technical issue is not drawing the tracker.

It is **resolving event identity and order reliably**.

You must handle:

* duplicate events
* corrected events
* delayed feed updates
* score corrections
* missing coordinates
* out-of-order websocket patches

So you need:

* `eventId`
* `sequence`
* `sourceUpdatedAt`
* idempotent inserts
* patch reconciliation

Example rule:
If same `eventId` returns with changed metadata, update existing event and recompute derived state.

Not doing that will make trackers buggy as hell.

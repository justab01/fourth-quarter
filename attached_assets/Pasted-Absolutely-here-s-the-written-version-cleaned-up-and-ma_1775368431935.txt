# FOURTH QUARTER — COMPLETE SYSTEM AUDIT

**Date:** April 5, 2026
**Platform:** Expo React Native (iOS/Android) + Express API

## 1. Executive Read

The app already has real bones. This is not some fake shell. The structure is strong, the ambition is strong, and the feature set is honestly deeper than a lot of early sports apps. You already have live scores, standings, player pages, team pages, draft center, articles, onboarding, AI summaries, and a multi-sport framework that goes beyond just NFL/NBA.

But deadass, the app has a few major fractures that will keep it from feeling premium until they get fixed:

* Search routing is the biggest structural problem
* Static data is still controlling too much of the experience
* Some features exist visually but are not fully real yet
* Spacing and layout rhythm are inconsistent across screens
* A few core states feel brittle or unfinished
* Personalization exists, but persistence and infrastructure are not fully there yet

So the truth is this:

**Fourth Quarter is closer to a real product than a concept, but it still feels like a powerful prototype instead of a polished sports platform.**

That is a good place to be. But now you need system cleanup, not more random feature stacking.

## 3. Architecture Audit

## What’s Strong

You have a legit app structure:

* Bottom-tab architecture is clean
* Dynamic routes for game, player, team, sport, article, and draft pages are smart
* Shared components and contexts are already in place
* API layer is separated from UI
* Multi-sport thinking is already baked into the product

That last part matters a lot.

You are not building one-off pages for every sport.
You are building a system that knows sports behave differently:

* team sports
* tennis
* combat
* multi-event

That is exactly the kind of thinking that can scale.

## What’s Weak

The architecture is good at the screen level, but not fully disciplined at the identity/data level.

That means:

* one player can route differently depending on source
* one team can exist in search but not fully exist in detail pages
* a page can look “built” but still be powered by partial or fake structure underneath

That’s the danger zone.
The UI is promising faster than the data model is stabilizing.

## 4. API + Data Layer Audit

## What’s Real and Good

Your app is pulling real information for:

* live scores
* schedules
* standings
* player stats
* career numbers
* game logs
* rosters
* articles
* draft data
* brackets

That is huge. That means the app already has real informational value.

## What’s Hurting You

The real issue is not that you lack live data.
The issue is that **static local scaffolding is still deciding too much of the experience**.

### Main Data Problems

1. **Only 13 “core teams” are truly mapped cleanly**

   * This is the biggest trust killer in the app
   * A user searches for a team, taps it, and gets a weak or broken result
   * Once that happens, the product feels fake even if the rest is real

2. **Team detail records are incomplete**

   * 90 teams exist in team detail data
   * but 77 are still basically skeletons
   * so coverage looks broader than it actually is

3. **Search index is ahead of actual page coverage**

   * you let users find things the app cannot truly support yet

4. **Hardcoded logic is still doing too much**

   * Houston teams
   * rival pairs
   * playoff month logic
   * these should eventually be computed or configurable

## What Needs to Happen

You need a cleaner hierarchy like this:

### Source of truth

* ESPN/live API first
* local static only as fallback
* never let static fake completeness

### One canonical ID format

For all teams and all players:

* one ID standard
* one route parsing standard
* one way to resolve detail pages

Right now your identity layer is split.
That is dangerous long term.

## 5. Search Audit — Biggest Problem in the Whole App

This is the most important section.

The search experience *looks* ambitious. That is good.
But under the hood, it is the most brittle part of the app.

## What Works

* full-screen modal feels modern
* grouped results are smart
* local + live search combo is good
* quick suggestion chips are useful
* debounce on live athlete search is smart

## What’s Broken

### 1. Dead Team Links

This is the loudest issue in the whole product.

You have 185 team entries in the search index, but only 13 truly mapped through `TEAM_NAME_TO_ID`.

That means:

* users can discover teams
* tap teams
* and land on pages that are incomplete, empty, or structurally wrong

That is unacceptable in a sports app.

Because in sports, users expect reliability.
You can’t tease discoverability and then fail resolution.

### 2. Player ID Inconsistency

Static players route one way.
Live ESPN athletes route another way.

That means the same class of object is being handled through different identities.
That creates:

* split logic
* incomplete data resolution
* weird bugs
* inconsistent page depth

### 3. Tennis/Individual Sports Confusion

ATP/WTA athletes being treated like “teams” is a structural design smell.

That tells me the search model is still too team-sport-first.

## Fix Priority

This is **Priority Zero**.

Before adding more features, fix:

* team ID resolution
* player ID normalization
* individual-sport entity modeling

Because if search is broken, the whole app feels unreliable.

## 6. Home Tab Audit

## What’s Good

The Home tab is trying to be a real command center, and that is the correct move.

It has:

* AI summary
* hero banner
* favorite-team content
* draft center
* watchlist
* daily game modules
* news clusters
* AI recaps

That’s strong product thinking.

## What’s Not Working

### 1. Too much dependency on content availability

Sections vanish if they are empty.
That causes vertical jumpiness.

A premium app should feel intentional even when content is missing.

### 2. The hero bleed hack is fragile

Using negative horizontal margin works until something changes globally.

This is the kind of thing that bites later.

### 3. Pull-to-refresh is incomplete

If “In One Breath” doesn’t refresh with everything else, manual refresh feels fake.

User thought process:
“I refreshed the app — why is this card still stale?”

That weakens trust.

## Recommendation

The Home page should become more systemized:

* fixed section shells
* predictable loading states
* complete refresh behavior
* less hardcoded editorial logic over time

## 19. Design System Audit

This is where a lot of your polish ceiling lives.

## Current Problem

The app has style.
But it does not yet have a fully disciplined visual system.

### Inconsistencies:

* screen padding: 16 vs 20
* section gaps: 4 vs 12 vs 14
* card radii: 14 vs 16 vs 18 vs 24
* hero edge bleed is hacked, not systemized

These things sound minor, but they are exactly the difference between:

* “this is cool”
  and
* “this feels premium”

## Recommendation

Lock a visual system now.

### Suggested baseline:

* horizontal padding: 16 everywhere
* section spacing: 12 or 16 consistently
* card radius: 14 standard
* hero radius: 20
* chip radius: 16 or full-pill depending on type
* one skeleton language across all tabs
* one empty-state language across all tabs

This one move alone will make the whole app feel way more expensive.

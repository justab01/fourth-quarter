System Audit – Fourth Quarter Mobile & API Platform (Expo React Native & Express, 5 April 2026)
Overview

This audit examines the Expo‑based Fourth Quarter mobile application (iOS/Android) and its Express API backend. The mobile codebase contains a rich set of screens (home, scores, sports, standings, news, search, profile, game detail, player detail, team detail, sport hubs, draft centre, articles, onboarding) and uses contexts and React Query for state management. The API surfaces 12 endpoints that proxy real sports data (live scores, standings, rosters, news, draft information, AI recaps, etc.) from ESPN, The Sports DB and OpenAI.

The audit below uses the user‑supplied architecture description, manual exploration of the preview app, and best‑practice guidelines from recognised design systems to identify issues and recommend improvements.

Screen‑by‑Screen Analysis
Home Tab

The Home tab greets the user and aggregates AI summaries, live games, favourite teams, draft centre cards, watchlists, “Today’s Games”, league overviews, top news and AI recaps. Padding is set to paddingHorizontal: 20 with gap: 4 between sections, and safe‑area insets adjust top/bottom spacing.

Observations:

The hero banner uses marginHorizontal: -20 to achieve edge‑to‑edge bleeding. This hack will break if horizontal padding is ever changed. Design systems like Atlassian recommend using a consistent base unit (8 px) and negative tokens or bleed primitives when content needs to extend beyond its container.
The pull‑to‑refresh control updates scores and news but does not refresh the “In One Breath” AI summary, leaving stale content.
Sections like “Tonight’s Watchlist” return null if empty, causing layout jumps. A consistent skeleton/placeholder should be shown instead, similar to other sections.

Recommendations: unify spacing and fix bleeding by adopting a design system with an 8‑pixel base unit; implement skeleton placeholders for empty sections; include AI summaries in refresh.

Scores Tab

The Scores tab contains a header, calendar strip, hero carousel, smart filters, league filter chips and league‑grouped game lists.

Issues & Recommendations:

Data is refreshed every 30 s; however, stale UI states occur when switching dates or filters. Implement skeleton loaders instead of empty spaces and ensure data clearing when leaving a date.
Filters (e.g., “My Teams”, “Rivalry”) rely on hard‑coded lists; these should be computed from the user’s favourite teams and rival data from the API.
The calendar strip scrolls ±14 days; allow users to type or pick a date beyond 2 weeks (e.g., selecting a specific game by date) and fetch from API accordingly.
Sports Tab

Displays quick link chips and a grid of sports cards with “X Live” badges or season phases. Cards use paddingHorizontal: 16 with gap: 12.

Issues: inconsistent horizontal padding compared with other screens (Home and Scores use 20 px). A unified spacing system using multiples of 8 px would ensure visual harmony across screens.

Standings Tab

Offers league tabs, tournament brackets (NCAA), hot/cold streak chips, zone legends and conference tables. Uses gap: 4 between sections and paddingHorizontal: 20.

Observations:

Some states use ActivityIndicator while others use skeleton loaders. Consistency across screens would improve perceived performance.
Expandable rows show “Why It Matters” explanations; verifying that this AI explanation is accurate and is refreshed regularly is important. Add a “Last updated” timestamp.
News Tab

Personalised news feed with filter chips, narrative clusters (Playoff Race, Injury Report, Trades), hero articles and lists. Uses gap: 4 between sections and gap: 28 between narratives.

Issues:

The News tab currently loads articles but there is no indicator of whether content is personalised (besides the tag). Provide user settings controlling league/team preferences and allow toggling between personalised and global news.
Refresh control uses a blue accent; unify refresh colours with the global accent (e.g., orange for NBA or team colour) for better branding.
Search Tab & Modal

Search triggers a full‑screen modal filtering static entries (1 655) and performing live ESPN athlete searches. Results are grouped (Teams, Athletes, Games, News).

Critical Issues:

Dead team links – TEAM_NAME_TO_ID contains only 13 core teams; other teams fallback to slugified names that may not match ESPN IDs, resulting in broken pages.
Inconsistent player IDs – static players use slugified names, whereas live ESPN results use {league}-{espnId}. This mismatch leads to different code paths; the static route does not support full career stats and news.
ATP/WTA players mis‑categorised – tennis players appear as “teams” in the search index and redirect to player pages using slugified names instead of {league}-{id}.

Recommendations:

Replace the static TEAM_NAME_TO_ID and TEAM_REGISTRY with a dynamic registry populated from ESPN’s team list endpoint. Provide fallback to the API when a static entry is missing.
Standardise ID formats across all routes (e.g., always use {league}-{espnId}) and deprecate slug‑based IDs. Create a slug alias mapping for deep links.
Update the search index to correctly classify individual sports players (ATP, WTA) as athletes rather than teams.
Profile Tab

The profile screen shows followed teams/leagues, AI insights (“Your go‑to league: NBA”), Fan/Nerd mode toggle, inline name editing, league/team toggles and notification settings. Dynamic theming applies team colours.

Issues & Recommendations:

Fan vs Nerd mode – verify that toggling the mode changes content density, data depth or analytics across screens (currently it might be purely cosmetic). Use the PreferencesContext to propagate the mode and conditionally render UI (e.g., hide advanced stats in Fan mode).
Follow button persistence – the “Follow” button on player pages only updates local state. Add an API endpoint (e.g., POST /api/user/favorites) to persist these preferences so that the user’s followed teams/players persist across devices.
Notification settings – there is a toggle in the profile, but no infrastructure exists for push notifications. Integrate with Expo’s Push Notification service or a third‑party provider to deliver live score updates, news alerts and draft notifications, and expose preferences in the API.
Game Detail

The Game detail page includes a hero section (scores and logos), floating tab bar (Gamecast, Box Score, Plays, Stats, Lineups), live tracker panel, SVG arena renderer, win probability bar, momentum graph, AI summary and real‑time WebSocket updates.

Observations:

The useGameSocket hook suggests WebSocket support, but server‑side events are not documented. Ensure the Express API has a WebSocket endpoint or use a service like Pusher for real‑time updates. Without server support, the hook may reconnect endlessly.
Live tracker and AI summary provide context. The AI summary should be generated after the game ends and integrated with a notification for followed teams.
For accessibility, include alt text for icons, ensure contrast ratio for the gradient backgrounds and provide haptic feedback for key interactions.
Player Detail

Six tabs: Overview, News, Stats, Bio, Splits, Game Log. Compact hero mode is used for the Game Log tab. Sport‑specific overview panels display different stat layouts; a milestone tracker shows progress bars toward career goals. The Game Log supports two modes: a sticky‑column table and a live event‑grouped display with filters (all, last 5, last 10, home, away, wins, losses).

Issues:

News & Splits tabs show “Coming Soon” – these should be implemented or hidden until ready.
Follow button is local state only (see profile). Tapping follow/unfollow should update the user’s preferences via API and reflect across the app.
Accessibility: ensure the sticky column is scrollable via keyboard and screen readers.
Team Detail

Draft leagues (NFL, NBA, NHL, MLB, WNBA) show seven tabs (Scores, News, Standings, Stats, Roster, Depth Chart, Future). Non‑draft leagues omit the Future tab. Static data merges with live rosters.

Data Gaps: 77 of 90 teams have skeleton data only. When data is missing, the page appears empty or shows placeholders.

Recommendations:

Use the ESPN API for all teams as a primary source; static records should only provide fallback fields like colours and logos. When a team is not fully supported, gracefully display “Information coming soon” rather than a blank page.
For MLS and soccer leagues, implement the “Transfers & Rumours” feature rather than a placeholder.
Ensure that the Future tab uses up‑to‑date draft picks and allows navigation back to the Draft Centre.
Sport Detail

Combines hero section, league filter chips, game list, top athletes, league links, schedule & results, news and standings CTA.

Recommendations: unify navigation between sport hubs and league hubs; provide quick filters to jump to major leagues; for multi‑league sports (football vs NFL, college football), clearly differentiate categories; implement “Live” badges using the API’s live game count.

Draft Centre

Three tabs: Pick Tracker, Prospects and Team Needs. League colours and emojis differentiate each sport. Issues: there are empty states (“No picks in Round X”) but no skeletons. The Team Needs tab might use outdated static lists; dynamic needs could be computed using current roster holes (e.g., PFF team needs for NFL). Prospects ranking should be updated from ESPN or other APIs.

Article Detail

Supports AI reading modes: Original, Easy, Quick, Cross‑Sport. Each mode uses OpenAI to rewrite articles with simplified language or cross‑sport metaphors. Provide haptic feedback on mode switch.

Issues: The article page lacks offline caching; reading modes should be persisted for returning to the same article; cross‑sport translation should respect user’s favourite sport.

Onboarding

A 5‑step wizard: Welcome → Pick Sports → Pick Teams → Name Entry → Confirmation. Saves to PreferencesContext. Ensure the onboarding flow persists when the app is force‑closed or interrupted; support editing preferences from the profile or settings.

Design & Spacing System

The project currently uses inconsistent padding (16 vs 20 px), varying gaps (4–12 px) and multiple border radii (14, 16, 18, 20, 24 px). Best‑practice design systems (e.g., Atlassian Design System) recommend adopting an 8‑pixel base unit and deriving spacing tokens from this base. This ensures that margins, padding and gaps scale consistently across different screen sizes. For instance, small gaps can be 8 px, medium gaps 16 px and large gaps 24 px.

Similarly, border radii should be consistent across components. The Telerik Design System notes that uniform corner radii create a cohesive appearance and warns against mixing rounded and sharp corners on the same element. Nested elements should use proportional radii that relate to their parent and maintain harmony. Where emphasis is needed (e.g., CTAs), a distinct radius may be used to signal primary actions.

Recommendations for Spacing & Layout
Unify horizontal padding – adopt a single value (e.g., 16 px or space.200/space.250) for all screens. Use the same base unit for left/right padding and adjust for safe‑area insets.
Standardise section gaps – choose a medium‑size token (12 – 16 px). For example, use 12 px between section headings and content and 16 px between major groups.
Standardise card radii – pick a default radius (e.g., 14 px) and a larger radius for hero banners (e.g., 20 px). Avoid mixing 16 px, 18 px and 24 px on similar components; maintain consistent radii across nested elements.
Replace negative margins – the Home hero uses marginHorizontal: -20 to bleed edges. Use a bleed/edge primitive or negative space token as recommended in design systems. This ensures the pattern adapts when global padding changes.
Skeleton states – implement skeleton placeholders for every list or section that fetches data. They provide a consistent perceived performance and avoid layout jumps.
Navigation & Routing

Working routes cover all pages, but search results route to broken pages for many teams/players. The Not Found page is minimal.

Recommendations:

Dynamic route resolution – centralise route parsing so both static and API‑based IDs resolve correctly. Use a single canonical ID format for teams and players and provide redirection from legacy slug formats.
Comprehensive team registry – populate the registry from ESPN’s master team list and update it during app startup or build time. Include league, abbreviations, logos, colours and current season meta. This will prevent “dead links” in search.
Individual sports (tennis/combat) – treat athletes as first‑class entities; create dedicated player routes for them rather than treating them as “team” pages.
Enhanced 404 page – include suggestions like top games, trending articles and a search field to help users recover from bad links.
Data Quality & Caching
Fallback strategy – when static data is missing or outdated, fallback to the live ESPN endpoint. Use a caching strategy similar to React Query’s stale/refresh parameters. Avoid hard‑coding rival pairs, playoff months and other dynamic values; compute them from standings and schedules.
AI Content freshness – ensure that AI summaries (In One Breath, AI recaps, Why It Matters) are refreshed based on game or article updates. Provide a timestamp to show when the AI message was generated.
User favourites – store followed teams/players on the server so that the user’s preferences persist across devices and sessions. Provide endpoints to update and retrieve preferences.

Conclusion

The Fourth Quarter platform delivers an ambitious multi‑sport experience, integrating live data, AI summaries, draft analysis and personalised feeds. However, the current implementation suffers from inconsistent spacing, static data limitations and routing issues. Adopting a design system with a consistent spacing scale (e.g., an 8‑pixel base unit) and consistent border radii, replacing static registries with dynamic sources, persisting user preferences, and completing missing features will significantly enhance usability and maintainability. Prioritising critical fixes around search routing and follow persistence will prevent broken user journeys and improve trust in the app.

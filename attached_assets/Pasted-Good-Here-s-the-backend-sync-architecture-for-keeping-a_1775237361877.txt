The simple system

Your app should run on 4 layers:

1. Ingest

Pull from real sources only:
	•	schedules
	•	teams
	•	players
	•	standings
	•	game stats
	•	playoff series schedules
	•	draft prospects / draft order / picks

Sportradar already exposes NBA schedules, playoff series schedules, player profiles, daily change logs, and draft/prospect feeds, and official league sites publish current draft centers and key dates.  ￼

2. Normalize

Convert every source into your own clean tables:
	•	teams
	•	players
	•	games
	•	game_logs
	•	competitions
	•	stages
	•	draft_prospects
	•	draft_picks
	•	team_draft_assets

3. Refresh

Run scheduled sync jobs:
	•	live games: fast
	•	standings/rosters: medium
	•	draft/prospect/order: event-based and seasonal
	•	historical corrections: daily change log / patch jobs

Sportradar’s NBA and NFL docs both highlight change-log based update workflows specifically for updates to teams, players, schedules, standings, and stats.  ￼

4. Serve

Your app reads from your database, not directly from the external feeds on every screen load. That gives you speed, stability, and control.

How the 3 features use that data

1. Game logs

The app should never build rows from fake JSON. It should read:
	•	one real player
	•	one real game
	•	one real stat record

Then the UI groups them by:
	•	regular season → month
	•	playoffs → round
	•	tournament/cup → stage

Sportradar’s NBA schedules documentation explicitly separates daily schedule, full season schedule, and series schedule for playoffs, which is exactly what your segmentation layer needs.  ￼

2. Postseason / championship segmentation

Do not hardcode “Finals” manually in the frontend.

Instead store:
	•	competition_type = playoffs
	•	stage_name = Conference Finals
	•	series_id = xyz

Then the frontend renders:
	•	Conference Finals
	•	NBA Finals
	•	Wild Card
	•	World Series
	•	etc.

For NBA specifically, playoff series schedules are available as a separate feed, which gives you cleaner stage handling than guessing from dates alone.  ￼

3. Draft center

Draft pages should read from:
	•	draft_prospects
	•	draft_picks
	•	team_draft_assets

For NFL and NBA, draft/prospect structures are already formalized in vendor and official league pages, including prospect endpoints, top prospect feeds, combine measurements, and official draft hubs.  ￼

Refresh schedule you should use

Live games

Run:
	•	scoreboard/game state sync every 15–30 seconds
	•	player box score sync every 30–60 seconds
	•	play-by-play more frequently later if you add it

Schedules / standings / rosters

Run:
	•	every 2–6 hours
	•	plus nightly full reconciliation

Draft / prospects / order

Run:
	•	daily in quiet periods
	•	every 1–2 hours during lottery/combine/draft weeks
	•	every 30–120 seconds during the live draft if you have a live source

That matters because official NBA and WNBA draft pages and official league draft centers change as dates, order, and event details update.  ￼

Historical corrections

Run:
	•	daily change-log job
	•	nightly repair job

Sportradar’s daily change-log pattern is built exactly for this kind of correction flow.  ￼

The anti-fake-data rules

These rules should be hardcoded into production.

Rule 1

If a player/prospect/team does not come from a verified source, do not show it.

Rule 2

If a feed is missing, show:
	•	“Data pending”
	•	“Official feed not available yet”
	•	“Draft order not finalized”

Never show synthetic fallback people.

Rule 3

Every entity must have:
	•	source ID
	•	source name
	•	last synced timestamp

Rule 4

Production should disable all mock records.

Use something like:

const ALLOW_SYNTHETIC_DATA = process.env.NODE_ENV !== 'production';

Then block any mock insert when production is live.

How to map the data cleanly

For NBA game logs

When a game comes in, assign:
	•	competition_type = regular_season if normal season
	•	competition_type = playoffs if series schedule / playoff context
	•	stage_name = First Round | Conference Semifinals | Conference Finals | NBA Finals
	•	month_bucket = April for regular-season grouping only

That gives you ESPN-style organization without hardcoding it in UI.

For drafts

When a prospect comes in:
	•	store real name
	•	store real source ID
	•	mark is_prospect = true
	•	connect to league + draft year

When a pick is announced:
	•	update draft_picks.prospect_id
	•	mark pick_status = selected
	•	attach team and trade note

Cache rules

Cache aggressively for:
	•	team pages
	•	standings
	•	prospect boards
	•	draft history

Cache lightly for:
	•	live scores
	•	on-the-clock draft screen
	•	live picks

Never trust cache forever

Every cached response should carry:
	•	fetchedAt
	•	sourceUpdatedAt
	•	staleAfter

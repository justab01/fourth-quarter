1. Ingestion Layer (the “adapters”)
Create one small background job per league:
	•	NFL Adapter → calls Genius Sports official feed every 15 seconds (live) + once per day (schedules/rosters)
	•	NBA Adapter → calls Sportradar or SportsDataIO
	•	Same for MLB, etc.
Each adapter normalizes the data into your schema (see below) and saves it.
2. Central Database (the single source of truth)
Use Postgres (or Supabase if you want zero-server hassle).
Core tables (simple & fast):
	•	teams (id, name, logo, league, division)
	•	players (id, name, team_id, position, height, weight, college, birthdate)
	•	games (id, home_team, away_team, date, status, score, play_by_play_json)
	•	rosters (team_id + season + full depth chart JSON)
	•	standings (season + conference/division + rank + W-L + GB + PCT)
	•	stats (player_id + season + all advanced stats as JSONB)
	•	news (title, summary, source, published_at) — pull from league RSS or paid wire
Add Redis for “today’s scoreboard” and “live games” so everything feels instant.
3. Internal Hub API (what your app + AI actually call)
Tiny REST endpoints your Expo app hits (and AI uses as context):
/hub/teams/{teamId}          → full team page data (Scores + News + Standings + Roster + Depth Chart)
/hub/teams/{teamId}/roster   → exact table you saw in ESPN screenshots
/hub/teams/{teamId}/standings → the 2025-26 division table (Knicks example)
/hub/players/{playerId}      → full player profile (bio + stats + game log + splits)
/hub/games/today             → live scoreboard
/hub/search?q=houston        → powers your global search
All responses are clean JSON — exactly what you need to render the Knicks standings page, Texans roster table, C.J. Stroud profile, etc.
